# SAP Fiori Elements on CAP - Technology Overlay

**Prerequisite files:** `cap-shared.md`, `ux-standards.md`, `sapui5.md` (for setup, installation, and blank-page rules)

**Scope:** Annotation-driven Fiori Elements apps

> **UUID keys in value help dialogs:** Use `ValueListParameterOut` (not `ValueListParameterInOut`) for the key mapping parameter to prevent the UUID ID from appearing as a visible column in the value help dialog. `ValueListParameterInOut` adds the key as both a visible and searchable column. Additionally add `@UI.Hidden: true` to the related entity's key field for full suppression:
> ```cds
> category_ID @Common.ValueList: {
>   CollectionPath: 'Categories',
>   Parameters: [
>     { $Type: 'Common.ValueListParameterDisplayOnly', ValueListProperty: 'name' },
>     { $Type: 'Common.ValueListParameterOut',         // [v] Out, not InOut
>       LocalDataProperty: category_ID, ValueListProperty: 'ID' }
>   ]
> };
> ```

> **CDS 9.x FK scalar annotation limitation:** FK scalars like `type_ID`, `owner_ID` (e.g. `genre_ID`, `author_ID`, `category_ID`) are generated by CDS and cannot be directly annotated in `.cds` files - the CDS compiler reports `Element "type_ID" has not been found`. Always annotate the **association** (e.g. `type`, `owner`) instead; the annotations propagate to the generated FK scalar in `$metadata`. Exception: when annotating at the service level (in the service `.cds` file), association annotations DO propagate correctly to FK scalars.

**Version note:** All findings below are verified against SAPUI5 1.136.7 unless otherwise noted.

## 0. Generation workflow - with and without Fiori Tools MCP

### With Fiori Tools MCP (preferred)

Fiori tools uses a **mandatory 3-step protocol**. Never skip a step or guess parameters - the tool will error.

```
Step 1: fiori_list_functionality(appPath)
  -> Returns a list of functionalityIds for the current project state
  -> Use only IDs from this list - never invent or guess a functionalityId

Step 2: fiori_get_functionality_details(appPath, functionalityId)
  -> Returns the required parameters and their valid values

Step 3: fiori_execute_functionality(appPath, functionalityId, parameters)
  -> Executes the operation with the exact parameters from Step 2
```

Common use cases:
- **Add a new page / floorplan**: `fiori_list_functionality` -> find the "Add Page" or "Generate" entry -> execute
- **Add an OData service**: same 3-step with the "Add OData Service" functionality
- **Add annotations**: same 3-step or edit annotation `.cds` files directly

Also call `ui5_get_guidelines` before any generation or modification, and `cds_search_model` for exact service paths.

> **Post-generation: strip `devDependencies` from the generated `package.json`.** `fiori_execute_functionality` generates an app `package.json` that includes `@sap-ux/eslint-plugin-fiori-tools` and `@ui5/cli` in `devDependencies`. When `cds-plugin-ui5` starts, it tries to resolve these via `@ui5/project` graph and fails - crashing the entire CAP server. After every `fiori_execute_functionality` call, open the generated app `package.json` and set `devDependencies` to `{}`. The app does not need these packages at runtime when served by `cds-plugin-ui5`.

### Without Fiori Tools MCP

1. Use `cds compile '*' --to serviceinfo` for service paths and entity names
2. Create `manifest.json` manually following §5 of this file
3. Create annotation `.cds` files manually following §3 of this file
4. Follow the `sapui5.md §11` post-generation checklist (including `ui5.yaml` library reconciliation)
5. Run `npx @ui5/linter` and validate `manifest.json` manually

## 1. When to choose Fiori Elements

Use FE when:
- The app fits one of the standard floorplans (List Report, Object Page, Analytical List Page, Overview Page, Worklist, Flexible Column Layout)
- Draft-enabled edit flows are needed (FE handles the entire lifecycle with zero controller code)
- FLP / SAP Launchpad integration is a first-class requirement
- Team expertise is in CDS annotations, not XML view development

For custom layouts, non-standard navigation, or heavy client-side interactivity, use Freestyle SAPUI5 instead.

## 3. Annotation patterns (verified working)

### 3.1 Essential structural annotations

```cds
// List display
@UI.LineItem: [
  { $Type: 'UI.DataField', Value: name },
  { $Type: 'UI.DataField', Value: status, Criticality: severityScore }
]

// Object Page header
@UI.HeaderInfo: {
  TypeName: 'Item', TypeNamePlural: 'Items',
  Title: { $Type: 'UI.DataField', Value: name },
  Description: { $Type: 'UI.DataField', Value: description }
}

// Sections and fields in OP form - FieldGroup MUST be a separate annotate statement
annotate service.Items with @(
  UI.Facets: [{
    $Type: 'UI.ReferenceFacet',
    Target: '@UI.FieldGroup#General',
    Label: 'General Information'
  }]
);
annotate service.Items with @(
  UI.FieldGroup #General: {
    Data: [
      { $Type: 'UI.DataField', Value: name },
      { $Type: 'UI.DataField', Value: status }
    ]
  }
);

// Filter bar
@UI.SelectionFields: [name, status, category_ID]
```

> **Action labels:** Bound and unbound actions must have `@title` or `@Common.Label` annotations. Without them, FE uses the raw CDS identifier as the button label (e.g. `startProgress` instead of "Start Progress").
>
> **Annotations on the service definition file may not propagate to FE button labels.** Always add action title annotations in the FE `annotations.cds` file using the `annotate service.Entity actions { }` form:
> ```cds
> // CORRECT: annotate the service entity actions block in annotations.cds
> annotate MyService.Items actions {
>   myAction   @title: 'Human Label';
> };
> ```
> Check every action in the service for a label annotation before delivery.

> **FK field text annotations - mandatory (BLOCKING) for every FK field in `@UI.LineItem`:**
> Every FK scalar field used in `@UI.LineItem` or `@UI.SelectionFields` MUST have:
> 1. `@Common.Text: association.displayProperty` to show the human-readable name instead of the raw key value
> 2. `@Common.Label: 'Human Name'` for the column header / filter label
>
> **This is a BLOCKING requirement.** Displaying raw FK IDs, UUIDs, or CUIDs to users is never acceptable - it is not a "known limitation". Fix it before delivery.
>
> Without `@Common.Text`, FE renders the raw UUID/key value in the column cell. Without `@Common.Label`, FE shows the raw field name (e.g. `genre_ID`) as the column/filter header.
>
> **Critical FE 1.136.7 limitation:** Adding `@Common.Text` on a **navigation association** (e.g. `author`) used in `@UI.LineItem` disables PropertyInfo validation for the entire table and silently drops all other columns. Always annotate the FK **scalar** field instead (e.g. `author_ID @Common.Text: author.name`), or expose a flat computed scalar (`authorName`) from the service projection.

> **UUID FK scalar in Object Page FieldGroup shows raw UUID:** If a category/type field in an Object Page shows a UUID string, annotate the **association** (not the scalar) with `@Common.Text` + `@Common.TextArrangement: #TextOnly` - the annotation propagates to the FK scalar in `$`metadata:
> ```cds
> annotate service.Items with {
>   category @Common.Text: category.name @Common.TextArrangement: #TextOnly;
> };
> ```
> This also prevents the UUID from appearing as a raw ID string in value help dialogs.

> **Integer fields representing years or codes - and integer business keys:**
> Any integer field used as a business key (e.g. `TravelID`, `IncidentID`, `OrderID`) or a calendar year MUST be annotated with `@Common.IsCalendarYear: true` to prevent FE from applying locale thousand-separator formatting (which turns `4143` into `4,143` or `1847` into `1,847`):
> ```cds
> TravelID   @Common.IsCalendarYear: true;  // e.g. - replace with your actual integer key field name
> IncidentID @Common.IsCalendarYear: true;  // e.g. - replace with your actual integer key field name
> publishedYear @Common.IsCalendarYear: true;
> ```
> This applies to any integer that is a code or identifier - not just actual calendar years. The annotation name is misleading but it is the correct mechanism to suppress thousand-separator formatting in FE.

> **Pitfall - `UI.FieldGroup#Qualifier` in a combined annotate block causes a CDS compilation error:**
> The `#` character is not a valid identifier key inside an `@(...)` block. Always place qualified annotations in their own separate `annotate` statement.
>
> ```cds
> // WRONG - CDS compiler error: Mismatched '=', expecting [Identifier], '}', '@'
> annotate service.Items with @(
>   UI.Facets: [...],
>   UI.FieldGroup#General: { Data: [...] }   // [v] # not valid as key here
> );
>
> // CORRECT - qualified annotation in its own statement
> annotate service.Items with @(
>   UI.Facets: [...]
> );
> annotate service.Items with @(
>   UI.FieldGroup #General: { Data: [...] }
> );
> ```

### 3.2 Value help (FK picker)

```cds
// Annotate both the FK scalar AND the value help on the FK scalar:
category_ID @Common.ValueList: {
  CollectionPath: 'Categories',
  Parameters: [
    { $Type: 'Common.ValueListParameterDisplayOnly', ValueListProperty: 'name' },
    { $Type: 'Common.ValueListParameterOut', LocalDataProperty: category_ID, ValueListProperty: 'ID' }
  ]
};
category_ID @Common.ValueListWithFixedValues: true;  // renders as dropdown, not dialog
```

> **Pitfall:** Annotating only the navigation association (not the FK scalar) has no effect. Always annotate the `_ID` field.

> **Pitfall:** `ValueListWithFixedValues: true` on a property that also has a qualified `@Common.ValueList#Qualifier` crashes the FE `FilterField` - remove `ValueListWithFixedValues` from any property with qualified VLs.

### 3.3 Draft enablement

```cds
service MyService {
  @odata.draft.enabled
  entity Items as projection on db.Items;
  // Composition children inherit draft - DO NOT add @odata.draft.enabled to children
}
```

After enabling draft, the list binding must filter `IsActiveEntity eq true` (FE handles this automatically). The FE generates Edit/Save/Discard buttons with no additional handler code.

**Suppress integer key dialog for auto-assigned keys** [verified-TrialJ-2026-05-27]:
Both annotations are required  -  either alone leaves the dialog visible:
```cds
annotate service.Items:ID with @Core.Computed @Core.ComputedDefaultValue;
```

**Suppress "Editing Status" auto-injected filter** [verified-TrialJ-2026-05-27]:
```cds
annotate service.Items with @Capabilities.NavigationRestrictions: {
  RestrictedProperties: [{
    NavigationProperty: DraftAdministrativeData,
    FilterRestrictions: { Filterable: false }
  }]
};
```

**Named success toasts and semantic URL routing**  -  requires SemanticKey [verified-TrialJ-2026-05-27]:
```cds
annotate service.Items with @Common.SemanticKey: [title];
// Without this: toast = "Object created". With it: toast = "Item 'My Title' created".
// Also controls URL hash format: semantic key value instead of technical key.
```

**Override Object Page "New Object" heading** [verified-TrialJ-2026-05-27]:
Use separate per-entity i18n files (NOT `{0}`  -  see §4 limitations):
```json
// In manifest.json - each Object Page target gets its own file:
"ItemsObjectPage": { "settings": { "enhanceI18n": "i18n/i18n-items.properties", ... } }
```
```properties
# i18n/i18n-items.properties
T_NEW_OBJECT=New Item
T_ANNOTATION_HELPER_DEFAULT_HEADER_TITLE_NO_HEADER_INFO=Unnamed Item
```

### 3.4 Field control (read-only, mandatory, conditional)

```cds
// Static read-only
someField @Common.FieldControl: #ReadOnly

// Dynamic: locked when isApproved=true, mandatory when false
someField @Common.FieldControl: { $edmJson: { $If: [{ $Path: 'isApproved' }, 1, 7] } }
// Values: 1=ReadOnly, 7=Mandatory, 3=Optional, 0=Inapplicable

// Field-level read-only (NOT @UI.UpdateHidden - that only works at entity level):
// Use @Common.FieldControl for per-field control
```

### 3.5 Conditional visibility and field control (Delete, Edit, Create, Criticality)

```cds
// Hide Edit button per-record:
@UI.UpdateHidden: { $edmJson: { $Path: 'isApproved' } }   // CORRECT for draft entities

// Hide Delete button per-record:
@UI.DeleteHidden: { $edmJson: { $Path: 'isLocked' } }

// Hide Create button (UI only - backend still accepts POST):
@UI.CreateHidden: true
```

> **Critical:** `@Capabilities.UpdateRestrictions.Updatable` with a dynamic path expression is silently ignored for `@odata.draft.enabled` entities. Use `@UI.UpdateHidden` instead.

> **Critical:** `@Capabilities.InsertRestrictions.Insertable: false` in SAPUI5 1.136.7 FCL LROP also blocks navigation from List Report to Object Page, not just the Create button. Use `@UI.CreateHidden: true` for button-only suppression.

### 3.5b Criticality - do NOT use `{= expr }` CDS expression syntax

The `{= ... }` dynamic expression syntax is **not** valid inside `UI.LineItem` DataField `Criticality` values in CDS annotations. Using it causes a CDS compilation error that crashes the entire CAP server on startup:

```cds
// [X] WRONG - {= } is XML view syntax, not CDS annotation syntax; causes server crash
{ $Type: 'UI.DataField', Value: status, Criticality: {= ${status} = 'N' ? 3 : 1 } }

// [OK] CORRECT - reference a stored integer element mapped from the status code
{ $Type: 'UI.DataField', Value: status, Criticality: criticalityCode }
```

To drive criticality from a status code field:
1. Add an integer element to the CDS projection: `criticalityCode: Integer`
2. Populate it in an `after('READ')` handler: `entity.criticalityCode = entity.status === 'N' ? 3 : entity.status === 'I' ? 2 : 1`
3. Reference it in the annotation: `Criticality: criticalityCode`

Or use a fixed enum where dynamic mapping is not needed: `Criticality: #Negative` / `#Critical` / `#Positive` / `#Neutral`

### 3.6 Currency / unit display

```cds
// Annotate per service entity - does NOT transfer across service boundaries
amount @Measures.ISOCurrency: currency_code;
```

> **Pitfall:** `@Measures.ISOCurrency` on a `select from` projection property is dropped from EDMX by the CDS compiler - annotate the base entity instead.

> **Known behavior (1.136.7):** In edit mode, a currency field renders as TWO separate inputs (amount + currency). This is expected FE behavior.

### 3.7 Selection fields and filter bar

```cds
@UI.SelectionFields: [status, category_ID, name]
// Use denormalized scalar properties - NOT association paths (e.g. category/name)
// Association paths in SelectionFields produce UUID tokens in the filter bar, not readable labels
```

For readable filter tokens from FK fields, add a denormalized computed field:
```cds
categoryName: String @readonly;  // Do NOT use @Core.Computed - it disables FE 1.136.7 PropertyInfo (see below)
// Populate via after('READ') handler
// Annotate @UI.HiddenFilter: true if not needed as a filter target
```

> **ALP crash - never place bare navigation FK keys in `UI.SelectionFields` for ALP apps:**
> In an Analytical List Page, placing `author_ID`, `category_ID`, or any raw FK field in `UI.SelectionFields` without complete `@Common.Label`, `@Common.Text`, and `@Common.ValueList` annotations causes a runtime crash: `TypeError: Cannot read properties of undefined (reading 'map') at _formatPropertyInfo`. The safe default is:
> - Use only scalar primitive properties (String, Integer, Boolean, enum code fields) that have `@title` or `@Common.Label` in `UI.SelectionFields` for ALP
> - For navigation-based filtering, annotate the navigation property itself (e.g. `category`) with a proper `@Common.ValueList`, not the raw `_ID` key
> - When in doubt, set `@UI.SelectionFields: []` (empty) - the ALP will show a filter bar with no default filters, which is always safe

> **ALP `views.paths` dual-view requires `$apply`-capable entity:**
> The `views.paths` configuration in `manifest.json` (chart + table dual-view selector) requires the bound entity to support OData `$apply` aggregation. Standard CAP projections do not support `$apply` unless annotated with `@Aggregation.ApplySupported`. Unless the entity is specifically designed as an aggregate, use a plain LROP (remove `views.paths`) - it always works. Document the ALP upgrade as a follow-up once a dedicated aggregate entity exists.

> **Auto-generated FK scalar labels (`flight_ID`, `connection_ID`, etc.):** When using entities with composite or association-based keys, the auto-generated FK scalar (e.g. `flight_ID`) inherits its filter bar label from the **association annotation**, not from the scalar itself. Always annotate the association with `@title`:
> ```cds
> annotate service.Items with {
>   flight @title: 'Flight ID';  // propagates title to flight_ID in filter bar
> };
> ```

### 3.8 Sorting and presentation

// Default sort (no qualifier = applies automatically to List Report):

| FCL 3-column composition child sub-OP | Shows "New Object" | Known 1.136.7 bug; no workaround |

| `@UI.Chart` in `@UI.HeaderFacets` | Not rendered | Headers support DataPoints only |
| `@Core.Computed: true` on projected scalars (e.g. CASE expressions) | **Disables PropertyInfo validation** for the entire table and filter bar - silently drops `@title`/`@Common.Label` on all FK scalars, causing filter bar labels to fall back to raw field names | Remove `@Core.Computed`; annotate scalars with `@readonly` only. `@Core.Computed: true` propagates to OData `$metadata` and is the root cause. |

Use FE 1.136.14+ or expose the entity through a non-draft read-only projection for the FE app. |
| FK scalar filter labels (FK scalars show raw field name, e.g. `type_ID` instead of "Type") | FE 1.136.7 does not consistently resolve `@title`/`@Common.Label` on FK scalars when PropertyInfo is disabled | Fix root cause (remove `@Core.Computed`). | Patch upgrades within `1.136.x` are not always safe | e.g. `1.136.17` introduced `DeleteEntry::ContextMenu is not defined in component manifest` absent in `1.136.7` | Before upgrading a patch version, test all pages and check console errors. Do not assume patch upgrades are non-breaking. |
| `DataPoint.MaximumValue` for Rating | Ignored - always 5-star | Hard limitation |
| `@Validation.Minimum/Maximum` on Integer | Not enforced | Use `@assert.range` instead |
| `@Common.IsUpperCase` | Not enforced | No FE enforcement |
| `CountRestrictions.Countable: false` | Does not suppress sub-table count | FE derives count from loaded data |
| `DataField.Criticality` in row popin (compact table) | No coloring | Use `@UI.Importance: #High` to keep in columns |
| `@UI.DataFieldAbstract.IconUrl` | Universally ignored | Not available |
| `@Capabilities.ChangeTracking` | No `$delta` queries | Metadata only |
| **`@Measures.ISOCurrency` embedded currency picker does not write back on Save** [verified-TrialJ-2026-05-27] | Changes made via the embedded currency widget inside the price input are NOT written back to the unit property on draft Save - cosmetic only | Remove `@Measures.ISOCurrency`; add the currency code as an explicit standalone field in the FieldGroup so users can change it directly |
| **Every List Report must have a paired Object Page target** [verified-TrialJ-2026-05-27] | FE 1.136.7 always configures `creationMode: "NewPage"` for any `sap.fe.templates.ListReport` regardless of `@Capabilities.InsertRestrictions`. | **`showDraftToggle: false` and `showShareButton: false` in LR target settings not recognised** [verified-TrialJ-2026-05-27] | The "Editing Status" combobox is auto-injected into every draft-enabled List Report filter bar and cannot be suppressed via these manifest keys | Annotate the entity with `@Capabilities.NavigationRestrictions: { RestrictedProperties: [{ NavigationProperty: DraftAdministrativeData, FilterRestrictions: { Filterable: false } }] }` - this is the only confirmed working approach |
| **`@Common.SemanticKey` required for named success toasts** [verified-TrialJ-2026-05-27] | Without it, all success toasts say "Object created" / "Object saved" regardless of `@UI.HeaderInfo.TypeName`; URL also uses technical key | Add `@Common.SemanticKey: [fieldName]` to the entity annotations (e.g. `[title]` for Books, `[name]` for Authors) |
| **`@Core.Computed: true` AND `@Core.ComputedDefaultValue: true` both required to suppress integer key dialog** [verified-TrialJ-2026-05-27] | Either annotation alone leaves the "Key (ID)" dialog visible on draft creation | Both must be present on the key field; `@Core.ComputedDefaultValue: true` alone is insufficient |
| **`sap.fe.core.ResourceModel.getProperty()` returns raw strings; no `{0}` substitution** [verified-TrialJ-2026-05-27] | `enhanceI18n` works but `T_NEW_OBJECT=New {0}` renders literally as "New {0}" - FE does not call `getText(key, [args])` on the enhanced bundle | Use static per-entity strings in separate i18n files, one per Object Page target: `"enhanceI18n": "i18n/i18n-authors.properties"` with `T_NEW_OBJECT=New Author` |

**Crash bugs - must avoid:**
- Duplicate property keys in `@UI.LineItem` - causes `PropertyHelper` crash
- `ValueListWithFixedValues: true` on a property with a qualified `@Common.ValueList#Qualifier`
- `@Aggregation.ApplySupported` in ALP-only annotation files (lost when ALP is stashed; crashes LROP)
- Entity-level `@Common.SideEffects` with virtual fields in `TargetProperties` - crashes `SideEffectsServiceFactory`

## 5. Manifest settings reference

### 5.1 List Report (LROP)

```json
"ListReport": {
  "type": "Component",
  "settings": {
    "entitySet": "Items",
    "initialLoad": "Auto",
    "variantManagement": "Page",
    "tableSettings": {
      "type": "ResponsiveTable",
      "selectionMode": "Multi",
      "enableExport": true,
      "condensedTableLayout": true
    }
  }
```

> **`initialLoad` values:** `"Enabled"` - table loads immediately on page open without any filter interaction (use for reporting/inventory apps where data should always be visible). `"Auto"` - loads only after the first filter interaction when `@UI.SelectionFields` are declared; behaves like `"Enabled"` only when no SelectionFields are present. `"Disabled"` (default) - user must press **Go** manually.
>
> **[verified-TrialJ-2026-05-27]** The descriptions of `"Auto"` and `"Enabled"` were previously inverted in this file. Confirmed correct behavior: always use `"Enabled"` when you want immediate auto-load. `"Auto"` does NOT auto-load when SelectionFields are declared.

### 5.3 Flexible Column Layout (FCL)

- Three routing targets with `controlAggregation`: `beginColumnPages`, `midColumnPages`, `endColumnPages`

### 5.5 Manifest rules verified in TrialJ [verified-TrialJ-2026-05-27]

**Rule: Every List Report must have a paired Object Page target, even for read-only reports.**
FE 1.136.7 always configures `creationMode: "NewPage"` for every `sap.fe.templates.ListReport`. Without a paired OP route and target, FE throws `creationMode NewPage navigation configuration missing` on load and leaves the toolbar broken.

```json
// CORRECT: always add a paired OP target even for a read-only BooksReport
{
  "pattern": "BooksReport({key}):?query:",
  "name": "BooksReportObjectPage",
  "target": "BooksReportObjectPage"
},
{
  "pattern": "BooksReport:?query:",
  "name": "BooksReport",
  "target": "BooksReport"
}
// ... in targets:
"BooksReport": {
  "name": "sap.fe.templates.ListReport",
  "options": { "settings": { "navigation": { "BooksReport": { "detail": { "route": "BooksReportObjectPage" } } } } }
},
"BooksReportObjectPage": {
  "name": "sap.fe.templates.ObjectPage",
  "options": { "settings": { "contextPath": "/BooksReport" } }
}
```

**Rule: `controlConfiguration` key must match the qualifier in `@UI.PresentationVariant.Visualizations`.**
Actions defined under `@com.sap.vocabularies.UI.v1.LineItem#Overview` are silently ignored if `@UI.PresentationVariant.Visualizations` references the unqualified `'@UI.LineItem'`. No error, no buttons. Always inspect `Visualizations` in annotations.cds and use the exact matching key (often unqualified):

```json
// If PresentationVariant.Visualizations: ['@UI.LineItem']  (unqualified)
"controlConfiguration": {
  "@com.sap.vocabularies.UI.v1.LineItem": {   // NO #qualifier
    "actions": { "GoToAuthors": { ... } }
  }
}
// If PresentationVariant.Visualizations: ['@UI.LineItem#Overview']
"controlConfiguration": {
  "@com.sap.vocabularies.UI.v1.LineItem#Overview": {
    "actions": { "GoToAuthors": { ... } }
  }
}
```

**Rule: To suppress Share button / Send Email in non-Launchpad environments:**
In a local CAP dev environment, Send Email is the only active share option. Setting `showSendEmail: false` removes it and collapses the entire Share button.
```json
"sap.fe": { "app": { "share": { "showSendEmail": false } } }
```

## 6. CAP annotation loading

Annotation files in `srv/annotations/` are **not** auto-loaded. They must be explicitly imported:

```cds
// services.cds or the service file
using from './annotations/items-annotations';
```

`@Capabilities.SearchRestrictions.Searchable: false` must be set in the service definition file - not in external annotation files.

`@Common.SemanticKey` defined in both a service file and an external annotation file: both appear in EDMX; the one appearing first in document order wins at runtime.

### Multiple FE apps targeting the same CAP service

When two or more Fiori Elements apps are generated for the same CAP service, their `annotations.cds` files are compiled together by CAP. This causes duplicate annotation errors if both apps define the same structural annotations.

**Rule:** When generating a second (or third) FE app for a service that already has an FE app:
- **Never** redefine `@UI.LineItem`, `@UI.HeaderInfo`, `@UI.PresentationVariant`, or `@UI.Facets` for an entity that another app has already annotated - CDS fails with `Duplicate assignment with "@UI.LineItem"` and the CAP server crashes on startup.
- The second app's `annotations.cds` should either be **empty** (relying on the first app's annotations), or add only non-conflicting annotations (e.g. `@UI.SelectionFields` with different fields, or `@Common.Text` on FK fields not yet annotated).
- Only one app per service should define global structural annotations (`HeaderInfo`, `LineItem`, `PresentationVariant`, `Facets`).

```cds
// [X] WRONG - second app redefines @UI.LineItem that the first app already defined
// This crashes CAP: [ERROR] Duplicate assignment with "@UI.LineItem"
annotate TravelService.Travels with @UI.LineItem: [...]

// [OK] CORRECT - second app's annotations.cds is empty or adds only non-conflicting items
// The first app's @UI.LineItem applies globally to all apps sharing the service
```

### Removing fiori-tools-appreload middleware (CAP multi-app environments)

The Fiori Tools generator adds `fiori-tools-appreload` middleware to `ui5.yaml`. This middleware uses a fixed port (35734) for live reload. When multiple FE apps are co-served on the same CAP instance, the second app's reload server fails with `EADDRINUSE: address already in use 127.0.0.1:35734`.

**Post-generation step for all FE apps served via `cds-plugin-ui5`:** Remove `fiori-tools-appreload` from `ui5.yaml`:

```yaml
# REMOVE this entire server middleware block from ui5.yaml after generation:
server:
  customMiddleware:
 - name: fiori-tools-appreload
      afterMiddleware: compression
      configuration:
        port: 35734
        path: webapp
```

`cds watch` handles reloading - the FE app does not need a separate live reload server when co-served with CAP.

## 7. First-attempt blank screen - FE-specific additions

For the full 5-stage blank screen diagnostic (Stages 1-5), see `sapui5.md §12`. The following are Fiori Elements-specific additions that apply at Stage 4 and Stage 5.

### FE-specific Stage 4 causes (app shell loads but no content)

| Symptom | Cause | Fix |
|---|---|---|
| App shell visible, no List Report content | Entity set name in manifest does not match `$metadata` exactly (case-sensitive) | Check `entitySet` in manifest against `GET $metadata`; e.g. `"Items"` != `"items"` |
| Spinner appears then disappears, empty page | `$metadata` request failed | Check Network tab for the `$metadata` request status |
| Blank page, `SideEffectsServiceFactory` error in console | Action-level `@Common.SideEffects` present | Remove immediately - fatal crash in 1.136.7 |
| Blank page, `PropertyHelper` error in console | Duplicate property key in `@UI.LineItem` | Remove the duplicate entry |
| App loads but List Report shows no filter bar and no table | `@UI.LineItem` and `@UI.SelectionFields` annotations not loaded | Check annotation file is explicitly imported in service `.cds` file |
| OVP cards all blank | `.v4` suffix missing from card template | `"template": "sap.ovp.cards.v4.list"` (not `"sap.ovp.cards.list"`) |

### FE-specific Stage 5 causes (table empty)

| Symptom | Cause | Fix |
|---|---|---|
| List Report loads, table shows "No items available", `initialLoad:"Enabled"` set | `cds deploy` not run after adding entity | `npx cds deploy --to sqlite` from CAP root |
| Table empty, Network shows 200 with empty `value: []` | Draft entity without `IsActiveEntity eq true` filter | FE handles this automatically - check `@odata.draft.enabled` is on the service entity (not DB entity) |
| Table loads but Edit button missing | `@odata.draft.enabled` not present on the service entity | Add annotation to the correct entity in the service file |
| Table loads, clicking a row does nothing (FCL) | `@Capabilities.InsertRestrictions.Insertable: false` applied | This blocks OP navigation - use `@UI.CreateHidden: true` instead |
| Object Page blank after row click | `HasDraftEntity` / `IsActiveEntity` not in key - using non-FCL routing | Ensure FCL routing is configured per `fiori-elements.md §5.3` |

### Checking `$metadata` directly

When annotations seem correct but nothing renders, fetch `$metadata` to confirm the annotation compiled:

```sh
curl "http://localhost:4004/<servicePath>/\$metadata" | grep -i "LineItem\|HeaderInfo\|SelectionField"
```

If the annotation name is absent from `$metadata`, the annotation file is either not imported or contains a CDS syntax error (check `cds watch` terminal for compile errors).

## 8. UX patterns specific to Fiori Elements

### 8.1 After a bound action - OP does not refresh automatically

After clicking a bound action, the Object Page does not refresh in SAPUI5 1.136.7. Users see stale data until navigating away and back. Mitigations:
- Use `req.notify()` in the CAP handler to emit a toast (confirms the action ran)
- Document to users that refreshing shows the updated result
- Place action results in a dedicated timestamp field that appears in the OData response

### 8.2 Multiple CAP messages from one action

`req.notify() + req.warn()` from one action -> single dialog, not multiple toasts. Severity hierarchy for dialog title: Error > Warning > Success > Information.

Design for actions:
- Single success + no caveats: `req.notify()` alone -> auto-dismiss toast
- Success + warning: `req.notify() + req.warn()` -> "Warning" alertdialog (user must close)
- Avoid sending informational messages users don't need to act on - they still block with a dialog

### 8.3 `req.warn()` in `before SAVE` - save completes before dialog is dismissed

`req.warn()` in a `before SAVE` handler fires AFTER the save commits. The warning dialog appears after the fact - users cannot cancel via the dialog. Use `req.reject()` if the save should be blocked; use `req.warn()` only for advisory messages where save should proceed.

### 8.4 `@assert.format` exposes raw regex to users

`@assert.format: /^[A-Za-z]+$/` generates error messages with the raw regex pattern. Always provide a custom message:

```cds
field @assert.format: /^[A-Za-z]+$/ @assert.format.text: 'Enter letters only (A-Z)';
```

### 8.5 Delete guard - avoid two-dialog anti-pattern

`req.reject()` in `before DELETE` fires after FE already showed a "Confirm delete?" dialog - users see two dialogs. Prefer `@UI.DeleteHidden: { $edmJson: { $Path: 'isLocked' } }` to prevent the attempt entirely.

### 8.6 `@requires` on CAP service blocks co-served Fiori Elements app in development

When a CAP service has a `@requires` annotation (e.g. `@requires: 'support'`, `@requires: 'authenticated-user'`), the Fiori Elements app will fail to load in development with HTTP 401 or 403. `cds-plugin-ui5` does not inject authentication headers - the browser page loads without credentials and receives an error before `$metadata` can be fetched.

**Options for development:**

1. **Temporarily change to `@requires: 'any'`** on the service during development (revert before production deployment).
2. **Configure mock credentials** in the project's `package.json` `cds` section so the CAP server accepts unauthenticated requests from `localhost`:
   ```json
   "cds": { "requires": { "auth": { "kind": "mocked", "users": { "alice": { "password": "alice", "roles": ["support"] } } } } }
   ```
   Then configure the browser session (or use a `.cdsrc.json` with `autoLogin`) to send credentials.
3. **Use an unauthenticated service** (e.g. a `ReportingService` with no `@requires`) as the FE data source; handle authenticated write operations via separate fetch calls.

> **Never** use `http://user:pass@host/` URL syntax to pass credentials in a browser context - the Fetch API rejects requests constructed from URLs containing inline credentials (`TypeError: Request cannot be constructed from a URL that includes credentials`).
4. **Add a dev-only annotation override file** that extends the service with `@requires: 'any'`. Create `srv/dev-auth-override.cds`:
   ```cds
   using from './my-service';        // replace with your actual service file name
   extend service MyService with @requires: 'any';  // replace with your service name
   ```
   This file should be excluded from production builds (add to `.cfignore` or similar). This pattern is the most reliable option when `autoLogin` does not work for browser sessions with co-served FE apps.

### 8.7 Wide-field layout in FE Object Page forms (ER-LAYOUT-7) `[verified-TrialK-2026-05-29]`

**Problem:** The FE `ColumnLayout` framework control decides its column count at runtime based on form element count and viewport width. For an Object Page section with 2-3 form fields it typically chooses `column-count: 3` on its inner `<dl>`, giving each label+field pair only ~249px regardless of the field's `String(N)` length in the CDS model. `@UI.MultiLineText: true` makes the field a textarea, but does not change its width allocation.

**Confirmed behaviour in FE 1.136.7:** `@UI.PresentationVariant`, FieldGroup field ordering, and `columnsL` manifest settings have NO effect on the ColumnLayout's runtime column-count decision. The only confirmed workaround is a CSS override scoped to the specific form section.

**Required workflow for long-text fields (String(N) with N >= 60, or name matching `/title|description|notes|remark/i`):**

**Step 1 - Isolate the wide fields in their own `ReferenceFacet`.**

Do NOT place a long-text field in the same `CollectionFacet` as short-text fields (codes, selects, dates). Create a dedicated facet:

```cds
annotate service.MyEntity with @(
  UI.FieldGroup #WideFields : {
    Data : [
      { $Type : 'UI.DataField', Value : title,       Label : 'Title' },
      { $Type : 'UI.DataField', Value : description, Label : 'Description' }
    ]
  },
  UI.Facets : [
    // Full-width section for title + description
    { $Type : 'UI.ReferenceFacet', ID : 'WideFieldsFacet', Label : 'Incident', Target : '@UI.FieldGroup#WideFields' },
    // Two-column section for short fields
    { $Type : 'UI.CollectionFacet', ID : 'Overview', Label : 'Overview',
      Facets : [
        { $Type : 'UI.ReferenceFacet', ID : 'GeneralFacet',  Label : 'General',  Target : '@UI.FieldGroup#General' },
        { $Type : 'UI.ReferenceFacet', ID : 'DetailsFacet',  Label : 'Details',  Target : '@UI.FieldGroup#Details' }
      ]
    }
  ]
);
```

**Step 2 - Override `column-count` in the app CSS.**

Even with a dedicated `ReferenceFacet`, FE may still choose `column-count: 2` or `column-count: 3` for that section. Override it with a scoped CSS rule in the app's `css/app.css` file (registered in `manifest.json` under `sap.ui5.resources.css`):

```css
/*
 * Force the "WideFieldsFacet" ColumnLayout to a single column so each
 * label+field pair spans the full section width.
 * The [id*=] selector matches the runtime ID generated by FE from the facet ID.
 */
[id*="WideFieldsFacet"] .sapUiFormCLContainerCont {
    column-count: 1 !important;
}

[id*="WideFieldsFacet"] .sapUiFormCLContainerCont .sapUiFormCLElement,
[id*="WideFieldsFacet"] .sapUiFormCLContainerCont .sapUiFormCLElementDummy {
    width: 100% !important;
    display: block !important;
}
```

Replace `WideFieldsFacet` with the `ID` value you used in the `@UI.ReferenceFacet` annotation.

**Step 3 - Annotate the description field with `@UI.MultiLineText`.**

CDS `@UI.MultiLineText: true` on a service-level annotation is sometimes silently dropped by CAP's EDMX generator. If the field does not render as a textarea after adding the CDS annotation, add a raw OData XML annotation file (`webapp/annotations/annotations.xml`) and register it in `manifest.json`:

```xml
<!-- webapp/annotations/annotations.xml -->
<Annotations Target="MyService.MyEntity/description">
  <Annotation Term="UI.MultiLineText" Bool="true"/>
</Annotations>
```

```json
// manifest.json dataSources
"myAnnotations": { "uri": "annotations/annotations.xml", "type": "ODataAnnotation" }
// reference it in mainService.settings.annotations: ["myAnnotations"]
```

**Step 4 - Allow the textarea to be resized by the user.**

The SAPUI5 `sap.m.TextArea` wrapper sets `overflow: hidden` on its parent, which prevents the browser's native resize handle from appearing. Add these rules to the same CSS file:

```css
/* Allow vertical resize on all TextArea controls in edit forms */
.sapUiForm textarea.sapMTextArea { resize: vertical !important; min-height: 4rem; }
.sapMInputBaseContentWrapper:has(textarea) { overflow: visible !important; }
/* Fallback for browsers without :has() (Firefox < 121) */
.sapMTextAreaInner { resize: vertical !important; min-height: 4rem; }
```

**Gate (run during Facet 1 review, edit mode):**
```js
evaluate_script("() => { const w = [...document.querySelectorAll('textarea,input')].filter(el=>el.offsetParent&&/title|descr|note|remark/i.test(el.id+' '+(el.placeholder||''))); return w.map(el=>({id:el.id.substring(0,50),w:Math.round(el.getBoundingClientRect().width),ok:el.getBoundingClientRect().width>=250})); }")
// All ok:true required. Any ok:false = defect.
```
